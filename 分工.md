# FindBud 团队分工

> 平台：宁波诺丁汉大学学生，MVP = 数学建模比赛搭子
> 技术栈：前端 React + Vite（网页）/ 后端 FastAPI / 数据库 PostgreSQL

---

## xyh（Windows）

**后端 + 数据库 + AI 提示词**
- AI 提示词设计：设计选择题题目，量化三大向量维度（技能向量/性格动能因子/绝对实力）
- 建立数据库：负责所有表的建表和初始化（`docs/schema.md` 为准）
- 后端 API（与 hlk 合作）：用户信息提交、AI 选择题生成、答案提交接口

---

## xzf（Windows）

**前端页面**
- `OnboardingPage.tsx`：6 项基础信息填写（姓名/性别/年级/专业/组队目标/是否长期组队）
- `PreQuestionPage.tsx`：机动前置问题（1-2 道选择题，根据组队目标分流）
- `AIQuestionPage.tsx`：AI 生成的选择题页面（向量量化阶段）
- `MatchResultPage.tsx`：匹配结果展示（Top 3 候选人卡片）

---

## hlk（Mac）

**匹配算法（后端）+ 等待页面（前端）**
- 后端匹配算法（与 xyh 合作）：实现"平行与正交的效用函数"，`run-match` 接口，结果查询接口
- `MatchWaitingPage.tsx`：匹配等待页面，轮询后端，完成后跳转结果页

---

## 开发顺序

**第一阶段（并行）**
- xyh：建好数据库表 + 基础 API 骨架（用户信息提交接口）
- xzf：搭好前端路由，实现 `OnboardingPage`（可用 Mock 数据先跑通）
- hlk：设计效用函数逻辑（可先在本地用 Python 脚本验证算法）

**第二阶段（并行）**
- xyh：完成 AI 提示词设计 + AI 选择题生成接口
- xzf：实现 `PreQuestionPage` + `AIQuestionPage`
- hlk：实现 `run-match` 后端接口 + `MatchWaitingPage` 前端

**第三阶段（联调）**
- 前后端接口对接
- xzf 实现 `MatchResultPage`，调用 hlk 的结果接口
- 整体流程跑通测试
