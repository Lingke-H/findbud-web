# Background

> 我正在构建一个在校园内寻找搭子（包括学习搭子、竞赛组队、旅游搭子等）的平台的项目，这个项目面向宁波诺丁汉大学的学生，我现在正在准备搭建搭子**匹配模块**，我需要你帮助我完成该模块的**代码编写任务**。

我们的匹配思路是，以找数学建模大赛队友为例，先收集**用户（记作A）**的**固定标签**（标签由我们自己给出，不需要你来确定，具体参阅“**tag.md**”文件），以及通过问一些问题的形式对其的**分类标签**进行打分（比如“建模手：70，编程手：30，论文手：40；军官：65，勇士：40，军师：80；参加比赛数量：5，获奖数：2，获奖欲望：90”），然后向数据库中其他用户的信息先进行**固定标签**匹配，得到一个满足**固定标签**要求的最小用户集合，然后再将A以及该子集中用户相应**分类标签**的评分输入一个**效用函数**（反映适配度，值越大匹配度越高）中，得到匹配度最高的**3个**用户，作为A的匹配结果。

# Tasks

1. 你现在仅需关注**匹配模块**的代码编写。模块的输入流是用户A的**分类标签**信息以及满足A**固定标签**要求的最小用户集合（包含这些用户的信息，下面以在此集合中的用户B为例）。
2. 需要按照***每一组（例如“技能向量”）*** **互斥标签**和每一个**独立标签**构建**子效用函数**，然后通过**加权和法**构建总效用函数。
3. 对于***每一组（例如"技能向量"）*** **互斥标签**，将该组内所有子标签的得分构成向量（例如技能向量 = [建模手得分, 编程手得分, 论文手得分]），计算用户A与用户B该组向量的**余弦相似度**，并以 `互补指数 = 1 - cosine_similarity(A向量, B向量)` 作为A和B在该组的***"互补指数"***。余弦相似度越低（即两向量越"正交"），说明两人在该组技能/性格上差异越大、互补性越强，互补指数越高。
4. 对**独立标签**进行预处理与相似度计算，统一逻辑为**差异越小、得分越高**，具体规则如下:

   - **参与过的比赛场次**：先进行非线性映射（映射函数为 logistic 函数，将场次压缩到 (0, 1) 区间），再以 `相似度 = 1 - |A_mapped - B_mapped|` 计算两人的相似度；
   - **是否获奖**：布尔值转为 0（未获奖）/ 1（获奖），以 `相似度 = 1 - |A - B|` 计算，两人同为获奖或同为未获奖均得满分（1.0），一有一无得 0 分；
   - **获奖欲望**：原始分值范围为 0~10，以 `相似度 = 1 - |A - B| / 10` 计算相似度。

   > 注意：以上每个独立标签的相似度得分，与互斥标签组计算出的互补指数**地位平等**，均作为独立的"指标"参与后续归一化与权重计算。考虑到代码可复用性，对于开发者未指明映射函数的独立标签，默认映射函数为 `y = x`（即不做映射，直接使用原始得分）。
   >
5. 将***每一组*** **互补指数**以及每一个**独立标签**的指标进行**归一化**，得到归一化后的**互补指数**及**独立标签**的指标（以下两者统称**“指标”**）。
6. 确定每一个指标的权重。使用**熵权法**确定**客观权重**，**主观权重**由开发者人为制定（在代码中以字典形式集中定义，方便后续调整）。最终权重 = 客观权重 × α + 主观权重 × (1 - α)，当前融合系数 **α 暂定为 0.5（即客观:主观 = 1:1）**。

   > ⚙️ **开发者注意**：请在代码中以具名常量定义融合系数（如 `OBJECTIVE_WEIGHT_RATIO = 0.5`）以及主观权重字典（如 `SUBJECTIVE_WEIGHTS = {"技能互补指数": 0.3, "性格互补指数": 0.2, ...}`），需要调整时只改这两处，无需修改计算逻辑。
   >
7. 使用**加权和**法构造**效用函数**，计算A和B配对的效用函数值。
8. 对每一个满足**固定标签**要求的最小用户集合中的用户重复上述步骤3~6，计算他们的效用函数值。在这个循环遍历的同时，收集每个用户的**“特长标签“**，即针对***每一组*** **互斥标签**，通过组内比较的方式得到该用户最擅长的方向（即**”特长标签“**，比如说B的”建模手“指标的值在”技能向量“组内最大，那么”建模手“就是”技能向量“组的”特长标签“）。
9. 选取出**3个**效用函数值最大的集合中的用户作为A的**初步配对搭子**。
10. 对这三个初步配对搭子的每一个**特长标签**进行检查，如果发现三者***某一组*** 的**特长标签**全部相同，则将效用函数最低的那个**初步配对搭子**剔除，换上集合中**效用函数值排名下一位的用户**。重复这个过程，直到三者***没有任何一组*** 的**特长标签**全部相同为止。

    > 🛡️ **容错机制**：若候选集合中已无更多用户可供替换（候选池已耗尽），则立即终止替换过程，返回当前最优的三人组合，不强行满足特长不重叠条件。代码中应在返回结果里附加一个标记字段（如 `diversity_guaranteed: false`），以便上层调用者或日志层感知匹配结果的多样性保障情况。
    >

> 总结一下在**匹配模块**中构建的子模块（函数）：
>
> 1. **接收输入流模块**（步骤1）：接收用户A的分类标签信息及候选用户集合；
> 2. **互补指数计算模块**（步骤3）：对每一组互斥标签，计算A与候选用户B的余弦正交性（`1 - cosine_similarity`），输出该组的互补指数；
> 3. **独立标签预处理与相似度计算模块**（步骤4）：对各独立标签进行映射（logistic 或 `y=x`）并计算差异相似度（`1 - |A-B| / max_value`）；未指明映射函数的标签默认使用 `y=x`；
> 4. **归一化模块**（步骤5）：将所有互补指数与独立标签相似度统一归一化到 [0, 1]；
> 5. **权重计算模块**（步骤6）：熵权法计算客观权重，结合 `SUBJECTIVE_WEIGHTS` 中的主观权重，以 `OBJECTIVE_WEIGHT_RATIO`（默认 0.5）融合；
> 6. **初步配对模块**（步骤8、9）：遍历候选集，计算效用函数值并排序，取 Top 3；
> 7. **最终确定模块**（步骤10）：检查特长标签多样性，若某组特长全部相同则替换；候选池耗尽时触发容错，附加 `diversity_guaranteed` 标记后返回。
